- **状态**：P3 已修

### B2 — useWorkspaceEvents 跨会话误刷新（P2）

- **发现方式**：R1 架构风险测试（qa-blackbox 编写）
- **根因**：`useWorkspaceEvents.ts` 中 `handleAcpResponse`、`handleGeminiResponse`、`handleCodexResponse` 三个 IPC handler 无 `conversation_id` 过滤，任意会话的 tool_call 都会触发当前会话 workspace `throttledRefresh()`
- **单元测试**：`useWorkspaceEvents.r1.dom.test.ts` — FAIL（修复前）→ PASS（修复后）6 个用例
- **修复文件**：`src/renderer/hooks/useWorkspaceEvents.ts`
  ```ts
  // 修复：每个 handler 首行加过滤
  if (data.conversation_id && data.conversation_id !== conversation_id) return;
  ```
- **状态**：P2 已修 ✅

### B3 — useMessageLstCache 取消竞态（P2，争议后仲裁保留）

- **发现方式**：arch R2 分析，qa-blackbox 尝试复现
- **根因**：`useMessageLstCache` 缺少 DB 查询取消机制，快速切换时旧会话消息可能在新会话渲染后写入
- **修复**：dev-2 添加 `cancelled` flag，加载前设为 false，切换/清理时设为 true，写入前检查
- **争议**：dev 运行 3000ms 等待测试，修复前后均通过 → `[DISPUTE]`
- **仲裁（用户）**：保留修复（防御性代码）；qa-blackbox 须重写 ≤150ms 窗口测试或 mock IPC 延迟
- **状态**：P2 修复保留，测试待补 🔄

### R2 确认为 UX Bug（P2）

- **arch 确认**：快速 A→B→C→D 四连跳时，A 的消息（30 条）短暂出现在 D 界面（0.1~0.5s 窗口）
- **根因**：同 B3，DB 查询无取消机制
- **状态**：已记录 bug-reports.md，待 B3 测试补齐后一并验证修复
